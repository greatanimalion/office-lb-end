将权限拆解为 **“系统级操作”** 和 **“内容级操作”**：

| 类别 | 操作项 | 说明 | 对应 ONLYOFFICE config |
| :--- | :--- | :--- | :--- |
| **内容级（编辑器内）** | `view` (查看) | 只读，不能改 | `edit: false, comment: false` |
| | `comment` (评论) | 可看可评，不能改正文 | `edit: false, comment: true` |
| | `edit` (编辑) | 可修改内容、保存 | `edit: true` |
| | `download` (下载) | 导出/另存为 | `download: true` |
| | `print` (打印) | 打印文档 | `print: true` |
| **系统级（文件夹/列表）** | `delete` (删除) | 彻底移除文件 | 后端 API 拦截，不在编辑器配置中 |
| | `move` (移动) | 移动到其他文件夹 | 后端 API 拦截 |
| | `share` (分享) | 生成外链或邀请他人 | 后端 API 拦截 |
| | `manage` (管理权限) | 修改其他人的权限 | 后端 API 拦截 |



严格遵循 **RBAC（基于角色的访问控制）** 和 **ACL（访问控制列表）** 的标准模型，将“权限主体（谁）”、“权限客体（对什么）”和“权限动作（做什么）”完全解耦。

这套设计会重点解决当前结构中的几个痛点：**权限继承复杂**、**无法区分“授权给个人”与“授权给链接”**、以及**文件夹权限向上递归查询性能差**。

### 🧱 核心表结构设计（共 7 张表）

#### 1. 用户与组（基础）
```sql
-- 用户表（不变）
CREATE TABLE users (
    id INTEGER PRIMARY KEY,
    name TEXT,
    email TEXT UNIQUE,
    role TEXT DEFAULT 'user' -- admin / user
);

-- 组表（独立出来）
CREATE TABLE groups (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL,
    creator_id INTEGER NOT NULL, -- 组创建者
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 组成员表（多对多）
CREATE TABLE group_members (
    id INTEGER PRIMARY KEY,
    group_id INTEGER NOT NULL,
    user_id INTEGER NOT NULL,
    role TEXT DEFAULT 'member', -- member / manager / creator
    joined_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(group_id, user_id)
);
```

#### 2. 资源与层级（核心难点：文件夹继承）
```sql
-- 文件夹表（支持无限层级）
CREATE TABLE folders (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL,
    parent_id INTEGER, -- 父文件夹ID，NULL表示根目录
    group_id INTEGER NOT NULL, -- 所属组
    path TEXT NOT NULL, -- 关键字段：路径枚举，如 "/1/2/3/"，极大提升继承查询性能
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 文档表（只存元数据，权限继承自文件夹）
CREATE TABLE documents (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL,
    folder_id INTEGER NOT NULL, -- 必须属于某个文件夹
    owner_id INTEGER NOT NULL,  -- 创建者
    size INTEGER,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (folder_id) REFERENCES folders(id) ON DELETE CASCADE
);
```
> **关键改进**：文档**不再直接存储 `permission`**，所有权限由文件夹和权限表共同决定。`path` 字段让查询父级权限变得极其高效（`WHERE path LIKE '/1/2/%'`）。

---

#### 3. 权限定义与分配（核心权限库）
这里我采用 **“权限模板” + “分配记录”** 的两层设计，避免在 `permissions` 表里重复存储权限字符串。

**权限模板表（定义系统有哪些操作）**
```sql
CREATE TABLE permission_definitions (
    id INTEGER PRIMARY KEY,
    code TEXT UNIQUE NOT NULL, -- 'view', 'edit', 'delete', 'share', 'manage', 'download'
    description TEXT,
    category TEXT -- 'content' 或 'system'
);
```

**权限分配表（ACL 核心）**
```sql
CREATE TABLE acl_entries (
    id INTEGER PRIMARY KEY,
    -- 授权主体（谁拥有权限）
    principal_type TEXT NOT NULL, -- 'user' | 'group' | 'link'
    principal_id INTEGER NOT NULL, -- 对应 users.id / groups.id / share_links.id
    
    -- 授权客体（对什么资源）
    resource_type TEXT NOT NULL, -- 'folder' | 'document'
    resource_id INTEGER NOT NULL, -- 对应 folders.id / documents.id
    
    -- 具体权限（位掩码存储，性能最优）
    permissions_mask INTEGER NOT NULL DEFAULT 0, -- 如 7 表示 view+edit+delete
    
    -- 元数据
    granted_by INTEGER, -- 谁授予的
    granted_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    expires_at DATETIME, -- 可选：临时授权过期时间
    
    UNIQUE(principal_type, principal_id, resource_type, resource_id)
);
```

> **说明**：`principal_type = 'link'` 时，`principal_id` 指向 `share_links` 表的 ID，完美解决了“链接分享”与“用户权限”同表存储的混乱问题。

---

#### 4. 链接分享（独立管理）
```sql
CREATE TABLE share_links (
    id INTEGER PRIMARY KEY,
    token TEXT UNIQUE NOT NULL,
    resource_type TEXT NOT NULL, -- 'document' | 'folder'
    resource_id INTEGER NOT NULL,
    created_by INTEGER NOT NULL,
    password TEXT, -- 提取码
    max_views INTEGER, -- 限制访问次数
    view_count INTEGER DEFAULT 0,
    expires_at DATETIME,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

---

### 🔗 权限继承与查询逻辑

有了上述结构，权限查询可以做到**O(1) 单次查询**（配合缓存），完全避免递归：

1.  **查询用户的所有权限**：
    ```sql
    SELECT * FROM acl_entries 
    WHERE (principal_type = 'user' AND principal_id = ?)
       OR (principal_type = 'group' AND principal_id IN (SELECT group_id FROM group_members WHERE user_id = ?))
       OR (principal_type = 'link' AND principal_id IN (SELECT id FROM share_links WHERE token = ?))
    ```

2.  **判断用户对文档是否有 `edit` 权限**：
    校验顺序：`文档级 ACL` → `文件夹级 ACL` → 父级文件夹（利用 `path` 字段用 `LIKE` 一次性查出所有上级）→ `组默认权限` → `公共资源`。

3.  **避开 ORM 陷阱**：直接用 SQL 的 `CASE WHEN` 或应用层代码组合，一次性计算出最终权限。

### 💎 重构前后对比

| 设计维度 | 原设计 | 重构后设计 |
| :--- | :--- | :--- |
| **权限存储** | `permissions` 表混用，字段冗余 | `acl_entries` 标准 RBAC + ACL |
| **链接分享** | 混在 `permissions` 表中 | 独立 `share_links` 表，清晰且便于管理 |
| **文件夹继承** | 需要递归查询父级 | `folders.path` 字段实现 O(1) 查询所有父级 |
| **权限类型** | 字符串 `'read'`/`'write'` 不易扩展 | 位掩码支持任意组合且扩展性强 |
| **过期权限** | 无 | `acl_entries.expires_at` 支持临时授权 |
| **权限来源** | 来源不明 | `principal_type` 明确区分用户、组、链接 |

这套设计是目前业界（如 Google Drive、Confluence）在中小型文档管理系统中的标准实践。它**彻底解耦了“身份”与“权限”**，使得新增一种权限（如“导出为PDF”）只需插入一条 `permission_definitions` 记录，而不需要修改任何业务代码。

## 🔄 三、数据同步机制（核心流程）

### 3.1 文件上传/更新时触发索引

当用户上传新文档或更新文档内容时，执行以下流程：

```javascript
// services/document.service.js
async function indexDocument(docId) {
    // 1. 从 sql.js 获取文档元数据
    const doc = db.get(`SELECT * FROM documents WHERE id = ?`, [docId]);
    
    // 2. 提取文件内容（纯文本）
    let content = '';
    try {
        const filePath = path.join(UPLOAD_DIR, doc.storage_path);
        content = await extractTextFromFile(filePath, doc.file_type);
        // 截断至 1MB，防止 MeiliSearch payload 超限
        if (content.length > 1000000) {
            content = content.slice(0, 1000000);
        }
    } catch (err) {
        console.error(`提取文件内容失败: ${docId}`, err);
        content = ''; // 即使提取失败，也索引元数据
    }
    
    // 3. 计算该文档的权限扁平化数据
    const permissions = await calculateDocumentPermissions(docId);
    
    // 4. 构造 MeiliSearch 文档
    const msDoc = {
        id: doc.id,
        title: doc.title,
        content: content,
        owner_id: doc.owner_id,
        allowed_group_ids: permissions.allowedGroupIds,
        explicit_user_ids: permissions.explicitUserIds,
        is_public: permissions.isPublic || false,
        created_at: new Date(doc.created_at).getTime(),
        file_size: doc.file_size || 0
    };
    
    // 5. 推送到 MeiliSearch
    const index = meiliClient.index('documents');
    await index.addDocuments([msDoc]);
}
```

### 3.2 权限计算函数（扁平化）

```javascript
// services/permission.service.js
async function calculateDocumentPermissions(docId) {
    const doc = db.get(`SELECT * FROM documents WHERE id = ?`, [docId]);
    
    // 1. 检查是否公共文档
    if (doc.is_public) {
        return { allowedGroupIds: [], explicitUserIds: [], isPublic: true };
    }
    
    // 2. 查询文件夹继承权限（从 folders 和 acl_entries 表）
    const folderId = doc.folder_id;
    const inheritedGroupIds = getFolderGroupPermissions(folderId); // 递归或通过 path 查询
    
    // 3. 查询显式授予的权限（permissions 表）
    const explicit = db.exec(`
        SELECT to_id FROM permissions 
        WHERE target_id = ? AND resource_type = 'document' AND permission LIKE '%view%'
    `, [docId]);
    const explicitUserIds = explicit.map(row => row.to_id);
    
    // 4. 拒绝优先：如果有 deny 记录，返回空权限（本函数不处理，由校验层判断）
    
    return {
        allowedGroupIds: inheritedGroupIds,
        explicitUserIds: explicitUserIds,
        isPublic: false
    };
}
```

### 3.3 权限变更时触发重新索引

当 `permissions` 或 `group_members` 表发生变化时，调用：

```javascript
// 当权限表变动时

}
```

## 🔍 四、搜索接口实现（带权限过滤）

```javascript
// routes/search.routes.js
router.get('/api/search', authenticate, async (req, res) => {
    try {
        const { q, limit = 20, offset = 0 } = req.query;
        const userId = req.user.id;
        const userGroups = req.user.groupIds || []; // 用户所属组列表
        
        // 构建权限过滤器
        const filterParts = [];
        
        // 1. 公共文档
        filterParts.push('is_public = true');
        // 2. 自己创建的
        filterParts.push(`owner_id = ${userId}`);
        // 3. 显式授权给当前用户
        filterParts.push(`explicit_user_ids = ${userId}`);
        // 4. 当前用户所在组有权限
        if (userGroups.length > 0) {
            filterParts.push(`allowed_group_ids IN [${userGroups.join(',')}]`);
        }
        
        // 组合成 OR 条件（满足任意一条即可）
        const filterString = filterParts.join(' OR ');
        
        // 执行搜索
        const index = meiliClient.index('documents');
        const result = await index.search(q, {
            limit: parseInt(limit),
            offset: parseInt(offset),
            filter: filterString,
            sort: ['created_at:desc'],
            attributesToHighlight: ['title', 'content'],
            highlightPreTag: '<em>',
            highlightPostTag: '</em>'
        });
        
        res.json({
            hits: result.hits,
            total: result.estimatedTotalHits,
            limit: result.limit,
            offset: result.offset
        });
    } catch (error) {
        console.error('搜索失败:', error);
        res.status(500).json({ error: '搜索服务异常' });
    }
});
```

## ⚡ 五、性能优化策略

### 5.1 大文件异步处理（避免阻塞）
使用 **Bull 队列** 将文件内容提取和索引任务放入后台：

```javascript
// queues/index.queue.js
const Queue = require('bull');
const indexQueue = new Queue('document indexing', { redis: { port: 6379 } });

// 入队
indexQueue.add({ docId: 123 });

// 消费者
indexQueue.process(async (job) => {
    await indexDocument(job.data.docId);
});
```

### 5.2 批量索引（减少网络往返）
```javascript
// 启动时批量同步
async function batchSyncAllDocuments() {
    const docs = db.exec('SELECT id FROM documents');
    const BATCH_SIZE = 50;
    for (let i = 0; i < docs.length; i += BATCH_SIZE) {
        const batch = docs.slice(i, i + BATCH_SIZE);
        await Promise.all(batch.map(doc => indexDocument(doc.id)));
        await sleep(1000); // 避免 CPU 过载
    }
}
```

### 5.3 内存限制配置
启动 Node.js 时增加内存上限：
```bash
node --max-old-space-size=4096 app.js
```

## 💎 八、总结

这套方案严格遵循我们此前讨论的 **“扁平化权限 + 搜索时强制过滤”** 原则，具有以下优点：

| 维度 | 解决方案 |
|------|---------|
| **权限继承** | 在索引时预计算 `allowed_group_ids`，不递归查询 |
| **显式授权** | 存入 `explicit_user_ids`，支持单个用户特殊权限 |
| **拒绝优先** | 由业务层处理，拒绝时直接不放权（不在 MS 索引中体现） |
| **大文件处理** | 异步队列 + 内容截断（1MB），防止内存溢出 |
| **组用户 ≤1000人** | 不展开用户 ID，仅存储组 ID，查询时匹配 |
| **文件数量 700-800个** | 一次全量索引在几秒内完成，后续增量更新 |
| **搜索速度** | 预期 < 20ms，用户体验极佳 |
