# 项目流程

1. 系统内自带系统管理员负责查看管理-所有用户和文档，组的情况，所有用户需要进行登录进入系统
2. 所有用户进入系统前，需要先进行注册，注册登录后为普通用户
3. 用户进入系统后具有三种基本功能1.上传删除私有文件（私有文件不涉及权限相关,不会被任何人访问`不包括系统管理员`）。
   2 创建组（创建组后，默认身份为组超管理员，可以邀请局域网内用户，并设置相关用户权限与身份，其中组内身份分为三种a组超级管理员b组管理员c组普通用户，以及对文档文档当问权限，文件夹权限）
   3 分享文件,分享时必须设置分享类型（a.分享指定局域网内用户(可以设置文件权限，有效期限制，访问次数限制)，b.分享到局域网公共空间，任何人可见）
4. 用户进入系统内可以同时进入多个组，但在同一时刻只能以一个组的身份存在，可以切换不同组身份进行操作
5. 用户若想上传文档到组内，必须先上传至私有空间，然后进入到组内指定文件夹下，点击上传文件后才能上传到组内指定文件夹下，并且受到组内文件夹的权限控制

# 权限相关
- 权限控制采用后端于onlyoffice服务器双重校验机制
permission:项信息：谁（用户，文件夹，组）对谁(用户，组)作用于什么(文档)产生产生什么权限(0x00000000)，作用时间（time）,密码（ps）,次数(count)
## 项目流程：
1. 用户将系列信息传入后端，后端生成相应的token后，同时将信息存入+数据库
2. 下次其他用户直接将token传入后端，后端解析token并校验数据并比对数据库。
3. 若校验成功则返回走正常校验逻辑（获取配置文档）
4. onlyoff服务器第三方进行相关黑盒校验
此过程中token加密采用AES256加密，再不考虑高并发情况下，采用此方案。

# #权限拦截的具体实现
 所有用户点击文档是会先请求文档配置config,再获取配置的过程中会对用户身份进行校验，具体校验流程如下
1. 判断当前文档是否是公共文档，若是，直接返回
2. 判断用户是否是管理员，若是，直接返回。若不是管理员，分条件，两种，一.用户为系统内用户(分为普通用户，组内用户)，二.用户为外用户（此时只进入链接权限判断），
3. 内用户：
- 判断用户与文档是否存在所属关系，若是，则直接返回 若没有进入下一流程
- 判断用户是否与文档具有权限关系（查询permission表获取）,若有，立刻进行权限校验，，校验通过则直接返回，否者返回失败

针对于被作用或着被使用物体，都有一个权限字符串，用于表示该物体的权限。documents与folder均已实现，

## 文件权限继承机制

- 在组内（一个组具有完整的文件系统），创建文件时必须为文件赋予关于文档的权限（默认权限为直接父级文件夹权限继承，并可以在此基础上修修改）
  文件夹下的所有文档受到直接父级文件夹权限的控制，并可以通过有具有`修改权限`的组管理员及以上人员修改针对与该文档的权限

## 公共空间

- 网内公共空间，任何人可见，只可以上传，查看，下载文件，

## 私有空间

- 用户在系统内创建的私有空间，只能被用户自己可见，不能被任何人访问`不包括系统管理员`,自己具有文件的一切权限

## 组内人员权限机制

所有对人员权限的修改分为两种情况--1.临时修改，2.永久修改

- 组超级管理员直接管辖所有组内用户（包括组超级管理员）并对任何文档及文件夹具有所有权限以及设置除自己之外的用户的权限，
- 组管理员管理所有组内普通用户，并对组内所有文档及文件夹具有管理权限
- 组管理员及以上可以对组内普通成员的权限进行修改，如添加普通成员A对指定目录下的指定文件夹下所有文档的权限。

## 普通人员访问文档的权限优先级

- 被赋予的临时或永久权限 > 文档权限 > 文件夹权限

# 版本控制相关

- 系统内除公共空间外，其他组内文件，私有空间文件全部具有版本控制功能
- 系统还原只还原版本内容，不对文件权限进行还原
文档版本默认从1开始，以后每次加1，若存在版本1，2，3，4，5，当重版本5切换到版本1后，在1的基础上修改会创建版本6
## 版本更新触发时间
- 当用户对具有回溯功能的文档进行修改后关闭，服务器自动保存，并创建一个最新的版本。

## 版本控制细节

- 用户在私有空间内编辑文档后，会默认为新文档的版本号加一。文档版本号从1开始，每次编辑保存文档后，版本号会加一，组内文档同样如此（此外组内添加了文档的审计功能），所有文件均可进行版本切换。
- 用户从私有空间分享某人文档后,分享权限若具有编辑文档的权限，则其他用户编辑后若分享着处于离线状态（或者为非最后一个关闭文档的用户），则会创建一个新的临时版本，所有分享者都基于此临时版本进行编辑，所有被分享者离线后，系统会创建新的版本并加入该文件的审计路线中，且会以消息的方式通知给分享者，下次同理。
- 用户从私有空间分享某人文档后,分享权限若具有编辑文档的权限，则其他用户编辑后若分享着处于在线状态，则会创建一个新的版本，系统会创建新的版本并加入该文件的审计路线中，下次同理。

# 审计相关

# 文件上传流程
- 用户必须先在私有空间上传，然后进入到组内指定文件夹下，点击上传文件后才能上传到组内指定文件夹下，并且受到组内文件夹的权限控制

## 文件上传细节
系统采用操作系统的文件节点机制，采用文档节点，版本节点，属性内容分为两类，一类为文件属性随版本号变化，另一类为不变属性。

- 版本回溯-不变属性：文档权限，文档创建时间等
- 版本回溯-变化属性：文档的标题，文档的文件类型，md5值，文件大小，文件路径等
  表如下
```js
export interface DocumentVersion {
  id: number
  title: string
  document_id: number//集合映射，多对一关系
  filename: string
  filepath: string
  fileSize: number
  v_number: number
  alter_by: number//修改的用户id
  created_at: string
}
export type OwnerType = 'group'|'user'|'folder'|'public'
export interface Document {
  id: number
  document_v_id: number//确定当前的文档版本
  created_by: number
  created_at: Date
  version_number: number//用于记录最新的文档版本，方便下一次修改创建的文档版本号加一
  locked_by?: number//文档被锁定时，locked_by为锁定用户的id
  status: string//文档状态，active为活跃，inactive为已删除
  owner_id: number
  owner_type: OwnerType//ownerID与ownerType对应，用于记录文档的归属关系
  locked: boolean
  permission: string
}
```
1. 用户上传文件流合并成功后，后端会立刻创建对应的文件节点（对应document表） 。
2. 文件节点创建完成后会立刻创建版本节点（对应document_version表）。
3. 立刻更新document表的document_v_id字段，将其指向新创建的版本节点。

# 共享细节
- 共享分为内共享和外共享两种情况，内共享必须注册该系统账户，外共享无需注册，依靠链接直接访问文档。

## 内共享
- 内共享分为人员共享或是组内共享
- 人员共享需要执行目标人员唯一id，组内共享需要指定唯一组id，
## 外共享
1. 当系统外的用户点击链接后，会立刻进入该文档的当前版本。
- 当链接中涉及协调编辑功能时，会进入多人协作编辑该文档，期间按照正常用户操作进行。

2. 当系统内用户在共享文档页面点击后，系统会存储链接，方便下次立刻打开。
